# 개발 계획 (PLAN): 반도체 시료 생산주문관리 시스템

> PRD.md의 기능/비기능 요구사항을 구현 가능한 단위로 쪼갠 단계별(Phase) 개발 계획.
> 각 Phase는 이전 Phase의 결과물 위에서 동작하며, 완료 시 PRD §9 성공 기준(Acceptance Criteria) 중 해당 항목을 충족해야 한다.

## 진행 원칙

- 각 Phase 완료 시 최소 1개 이상의 의미 있는 커밋 생성 (PRD §7 Commit 이력 요구사항).
- 핵심 비즈니스 로직(수율/재고/상태 전이 계산)은 Phase 완료 시점에 단위 테스트를 함께 작성.
- 기존 PoC(`poc-console-mvc`, `poc-data-persistence`, `poc-data-monitor`, `poc-dummy-data-generator`)에서 검증된 구조를 최대한 재사용.

---

## Phase 0 — 프로젝트 뼈대 및 영속성 계층 구축

**목표**: MVC 패키지 구조와 SQLite 영속성 계층을 프로젝트에 이식하여 이후 모든 기능의 기반을 마련.

- `model/` `view/` `controller/` 패키지 구조 생성 (`poc-console-mvc` 구조 확장)
- SQLite 스키마 정의: `samples`, `orders`, 생산 큐(대기열) 저장 방식 결정 (별도 테이블 또는 `orders.status='PRODUCING'` + 메타데이터 컬럼으로 표현할지 결정)
- DB 연결/초기화 모듈(`db.py`) 작성 (`poc-data-persistence` 재사용)
- Sample/Order 엔티티 클래스 정의

**완료 기준**: 빈 DB로 앱을 실행/종료 후 재실행해도 스키마가 유지됨 (PRD §9 "재시작 후 데이터 유지"의 기반 검증).

## Phase 1 — 시료 관리 기능

**목표**: PRD §6.2 구현.

- 시료 등록 (ID/이름/평균 생산시간/수율 입력, 중복 ID 검증)
- 시료 조회 (전체 목록 + 현재 재고)
- 시료 검색 (이름 등 속성 검색)

**완료 기준**: PRD §9 "시료 등록 후 목록/검색에서 즉시 조회 가능" 충족. 등록/조회/검색 로직 단위 테스트 작성.

## Phase 2 — 시료 주문(예약) 기능

**목표**: PRD §6.3 구현.

- 주문 입력(시료 ID/고객명/주문 수량) 및 유효성 검증 (미등록 시료 ID 거부)
- 주문번호 채번 규칙 정의 (예: `ORD-YYYYMMDD-NNNN`)
- 주문 생성 시 상태 `RESERVED`로 저장

**완료 기준**: 예약된 주문이 DB에 `RESERVED` 상태로 저장되고 이후 승인/거절 대상 목록에 노출됨.

## Phase 3 — 주문 승인/거절 및 재고 판정 로직

**목표**: PRD §6.4, §8(재고 충분 판정, 부족분 계산) 구현.

- `RESERVED` 주문 목록 조회
- 재고 판정 로직: `재고 수량 ≥ 주문 수량` → 충분 / 아니면 부족
- 승인 처리:
  - 재고 충분 → 재고 차감 + 상태 `CONFIRMED` 전환
  - 재고 부족 → 부족분 계산 후 생산 큐 등록 + 상태 `PRODUCING` 전환
- 거절 처리: 상태 `REJECTED` 전환 (재고 변동 없음)

**완료 기준**: PRD §9 "재고 충분/부족 시나리오별 상태 전이" 두 항목 모두 충족. 재고 판정/분기 로직 단위 테스트 작성.

## Phase 4 — 생산 라인 (생산 큐 처리)

**목표**: PRD §6.6, §8(실 생산량/총 생산 시간 계산, FIFO) 구현.

- 생산 큐 자료구조 (FIFO) 및 조회 기능 (대기 중인 주문 목록)
- 실 생산량 계산: `ceil(부족분 / 수율)`
- 총 생산 시간 계산: `평균 생산시간 * 실 생산량`
- 생산 현황 표기 (현재 처리 중인 주문 정보, 진행 상태)
- 생산 완료 처리: 재고에 실 생산량 반영, 대상 주문 상태 `PRODUCING` → `CONFIRMED` 전환, 큐에서 제거 후 다음 항목 진행

**완료 기준**: PRD §9 "생산 완료 시 재고 반영 및 상태 전환" 충족. `ceil` 계산/FIFO 큐 동작 단위 테스트 작성.

## Phase 5 — 출고 처리

**목표**: PRD §6.7 구현.

- `CONFIRMED` 상태 주문만 출고 대상으로 목록화
- 출고 실행 시 상태 `RELEASE` 전환

**완료 기준**: PRD §9 "CONFIRMED 주문만 출고 가능, 처리 후 RELEASE 전환" 충족.

## Phase 6 — 모니터링

**목표**: PRD §6.5 구현.

- 주문량 확인: 상태별(`RESERVED`/`CONFIRMED`/`PRODUCING`/`RELEASE`) 건수·목록, `REJECTED` 제외
- 재고량 확인: 시료별 재고 수량 + 상태(여유/부족/고갈) 표기 로직

**완료 기준**: PRD §9 "모니터링 메뉴 정확성" 충족. 재고 상태 판정(여유/부족/고갈) 단위 테스트 작성.

## Phase 7 — 메인 메뉴 통합

**목표**: PRD §6.1 구현. Phase 1~6에서 만든 기능을 하나의 콘솔 앱으로 통합.

- 메인 메뉴 화면 및 요약 정보(등록 시료 수, 총 재고, 전체 주문 수, 생산라인 대기 수) 표시
- 각 하위 메뉴로의 라우팅 (Controller 계층에서 처리)
- 종료 처리 및 예외/입력 검증 공통화

**완료 기준**: 메인 메뉴에서 시료 관리 → 주문 → 승인/거절 → 생산 라인 → 출고 처리 → 모니터링까지 전체 플로우가 끊김 없이 동작.

## Phase 8 — 운영 도구 연동 (DataMonitor / DummyDataGenerator)

**목표**: 기존 PoC 도구를 본 프로젝트 DB와 연동.

- `poc-data-monitor` 로직을 참고하여 본 프로젝트 DB를 실시간 조회할 수 있는 관리자 도구 통합(또는 `--db` 인자로 기존 PoC 도구 재사용)
- `poc-dummy-data-generator` 로직을 참고하여 본 프로젝트 스키마에 맞는 더미 시료/주문 데이터 생성 스크립트 통합

**완료 기준**: 더미 데이터 생성 → 모니터링 도구로 실시간 조회까지 엔드투엔드 확인.

## Phase 9 — 테스트 정비 및 회귀 검증

**목표**: PRD §7 Test 요구사항 충족.

- Phase별로 작성된 단위 테스트를 정리/보강 (수율 계산, 재고 판정, 상태 전이, FIFO 큐, ceil 계산)
- 주요 시나리오(정상 흐름, 재고 부족 흐름, 거절 흐름)에 대한 통합 테스트 추가

**완료 기준**: PRD §9 "핵심 로직 단위 테스트 통과" 충족.

## Phase 10 — Clean Code 정리 및 문서 마감

**목표**: PRD §7 Clean Code, 문서 관리 요구사항 마감.

- 계층 간 책임 분리 재검토 (Model/View/Controller 경계 위반 여부 점검)
- 매직 넘버/중복 로직 제거, 네이밍 정리
- `CLAUDE.md`, `PRD.md`, `PLAN.md` 최신화 (구현 중 발견된 변경 사항 반영)
- 최종 커밋 이력 점검

**완료 기준**: 전체 PRD §9 Acceptance Criteria 전 항목 충족 확인.
