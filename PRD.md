# PRD: 반도체 시료 생산주문관리 시스템 (SampleOrderSystem)

## 1. 개요

### 1.1 배경

가상의 반도체 회사 **S-Semi**는 다양한 종류의 반도체 시료(Sample)를 생산하여 연구소, 팹리스(Fabless) 업체, 대학 연구실 등의 고객에게 납품한다. 시료는 주문이 들어오면 웨이퍼 공정 설비를 통해 제작되고, 검수를 거쳐 고객에게 출고된다.

최근 주문량이 급증하면서 엑셀·메모장 기반의 수작업 관리로는 다음과 같은 문제가 발생했다.

- 주문 처리 여부를 파악하기 어려움
- 공정 예약 후 완료 시점을 추적할 수 없음
- 재고가 충분한데도 불필요한 추가 공정이 진행됨

이를 해결하기 위해 콘솔 기반의 **"반도체 시료 생산주문관리 시스템"**을 개발한다.

### 1.2 목표

- 시료 등록/조회/검색, 주문 접수/승인/거절, 생산 라인 관리, 출고 처리를 하나의 콘솔 애플리케이션에서 일관되게 처리
- 상태 기반(State Machine) 주문 관리로 담당자가 주문/재고/생산 현황을 한눈에 파악
- 재고 부족 시 자동으로 생산 큐에 등록하여 생산 담당자의 수작업 개입을 최소화
- 데이터 영속성 확보(SQLite)로 애플리케이션 재시작 후에도 데이터 유지

### 1.3 범위 밖 (Out of Scope)

- 웹/GUI 화면 (콘솔 CLI로 한정)
- 고객의 이메일 요청 자동 수집/파싱 (주문 담당자가 수동으로 주문서를 생성하는 것을 전제)
- 다중 생산 라인 병렬 처리 (본 과제는 단일 생산 라인 기준)
- 사용자 인증/권한 관리 (역할 구분은 개념적 흐름 설명용이며, 시스템은 단일 운영자가 전체 메뉴에 접근 가능)

## 2. 사용자 및 역할

| 역할 | 설명 | 시스템 내 행위 |
|---|---|---|
| 고객 | 시료 요청자 (이메일 등 시스템 외부 채널로 요청) | 시스템에 직접 접근하지 않음 |
| 주문 담당자 | 요청에 맞게 주문서 작성 및 관리 | 시료 예약(주문 접수) |
| 생산 담당자 | 개발 시료 등록, 주문 승인/거절, 생산 관리 | 시료 등록, 주문 승인/거절, 생산 라인 조회, 출고 처리 |

> 본 시스템은 단일 콘솔 프로그램으로 구현되며, 메인 메뉴를 통해 위 역할의 모든 기능에 접근할 수 있다 (역할별 로그인 분리는 범위 밖).

## 3. 시스템 개요

- **생산 라인**: 공장에서 시료 하나를 생산하는 설비 흐름. 단일 라인이며 시료를 한 번에 하나씩 순차 생산. 주문이 들어온 시료에 대해서만 생산.
- **동작 방식**: 콘솔 기반. 담당자가 메뉴 번호를 입력하여 시료 등록 및 주문 처리를 수행.
- **아키텍처**: MVC(Model-View-Controller) 패턴 적용 (`poc-console-mvc` PoC에서 구조 검증 완료).
- **데이터 저장**: SQLite 기반 영속성 (`poc-data-persistence` PoC에서 CRUD 검증 완료).

## 4. 데이터 모델

### 4.1 Sample (시료)

| 필드 | 타입 | 설명 |
|---|---|---|
| `sample_id` | TEXT (PK) | 시료 고유 ID (예: `S-001`) |
| `name` | TEXT | 시료명 |
| `avg_production_time` | REAL | 평균 생산시간 (min/ea) |
| `yield_rate` | REAL | 수율 (0.0 ~ 1.0). 정상 시료 수 / 총 생산 시료 수 |
| `stock_quantity` | INTEGER | 현재 재고 수량 |

### 4.2 Order (주문)

| 필드 | 타입 | 설명 |
|---|---|---|
| `order_no` | TEXT (PK) | 주문 번호 (예: `ORD-20260416-0043`) |
| `sample_id` | TEXT (FK → Sample) | 주문 대상 시료 |
| `customer_name` | TEXT | 고객명 |
| `quantity` | INTEGER | 주문 수량 |
| `status` | TEXT | 주문 상태 (§5 참조) |
| `created_at` | TEXT | 주문 접수 일시 |

### 4.3 생산 큐 (Production Queue)

- 생산 라인 대기열. `PRODUCING` 상태로 전환된 주문이 등록되는 FIFO 큐.
- 큐 항목: `order_no`, `sample_id`, `부족분(shortage)`, `실 생산량`, `총 생산 시간`, `등록 시각`

## 5. 주문 상태(Order Status) 정의

| 상태 | 의미 |
|---|---|
| `RESERVED` | 주문 접수 (예약) |
| `REJECTED` | 주문 거절 — 정상 흐름 외 상태이며 모니터링 대상에서 제외 |
| `PRODUCING` | 주문 승인 완료, 재고 부족으로 생산 중 |
| `CONFIRMED` | 주문 승인 완료, 출고 대기 중 (재고 충분) |
| `RELEASE` | 출고 완료 |

### 상태 전이

```
RESERVED --(거절)--------------------------------------> REJECTED
RESERVED --(승인, 재고 충분)--> CONFIRMED --(출고 처리)--> RELEASE
RESERVED --(승인, 재고 부족)--> PRODUCING --(생산 완료)--> CONFIRMED --(출고 처리)--> RELEASE
```

## 6. 기능 요구사항

### 6.1 메인 메뉴

전체 시료 요약 정보(등록 시료 수, 총 재고, 전체 주문 수, 생산라인 대기 수 등)를 함께 표시하며 아래 5개 메뉴로 진입한다.

1. 시료 관리
2. 주문 (접수/승인/거절)
3. 모니터링
4. 출고 처리
5. 생산 라인

### 6.2 시료 관리

- **시료 등록**: `시료 ID`, `이름`, `평균 생산시간`, `수율`을 입력받아 신규 시료 추가. 등록된 시료만 주문 가능.
- **시료 조회**: 등록된 모든 시료 목록과 현재 재고 수량을 함께 표시.
- **시료 검색**: 이름 등 속성으로 특정 시료 검색.

### 6.3 시료 주문 (예약)

- 고객 요청에 따라 주문 담당자가 주문을 생성.
- 입력값: `시료 ID`, `고객명`, `주문 수량`.
- 생성 즉시 상태는 `RESERVED`.

### 6.4 주문 승인/거절

- **접수된 주문 목록**: `RESERVED` 상태 주문만 표시.
- **주문 승인**: 재고 상황에 따라 자동 분기.
  - 재고 충분 → 즉시 `CONFIRMED`로 전환.
  - 재고 부족 → 생산 라인(생산 큐)에 자동 등록, 상태를 `PRODUCING`으로 전환.
- **주문 거절**: 즉시 `REJECTED`로 전환.

### 6.5 모니터링

- **주문량 확인**: 상태별(`RESERVED`/`CONFIRMED`/`PRODUCING`/`RELEASE`) 주문 수 및 목록. `REJECTED`는 제외.
- **재고량 확인**: 시료별 현재 재고 수량과 상태 표기.
  - **여유**: 주문 대비 재고 충분
  - **부족**: 주문 대비 재고 수량 부족
  - **고갈**: 재고 수량이 0

### 6.6 생산 라인

- 주문량 대비 부족분을 수율과 오차를 고려하여 생산.
  - **실 생산량** = `ceil(부족분 / 수율)`
  - **총 생산 시간** = `평균 생산시간 * 실 생산량`
- 생산 완료 시 해당 주문 상태를 `PRODUCING` → `CONFIRMED`로 전환하고, 생산된 수량만큼 재고에 반영.
- **생산 현황 표기**: 현재 생산 중인 시료 정보(주문 정보, 진행률/생산량 등) 표시.
- **대기 주문 확인**: 생산 큐(FIFO)에 대기 중인 주문 목록 표시.

### 6.7 출고 처리

- `CONFIRMED` 상태 주문만 출고 가능.
- 특정 주문을 선택해 출고 실행 → 상태 `RELEASE`로 전환.

## 7. 비기능 요구사항

- **아키텍처**: MVC 패턴(Model/View/Controller 패키지 분리), `poc-console-mvc` 구조를 확장하여 적용.
- **데이터 영속성**: SQLite. 애플리케이션 재시작 후에도 시료/주문/생산 큐 데이터 유지 (`poc-data-persistence` 구조 확장).
- **데이터 모니터링 도구 연동**: `poc-data-monitor`에서 검증한 실시간 조회 방식을 활용해 운영 중 DB 상태를 확인할 수 있어야 함.
- **Dummy 데이터 생성 도구 연동**: `poc-dummy-data-generator`에서 검증한 방식으로 테스트용 시료/주문 데이터를 생성할 수 있어야 함.
- **Clean Code**: 계층 간 책임 분리(Model=데이터/도메인 로직, View=콘솔 입출력, Controller=흐름 제어), 매직 넘버·중복 로직 최소화.
- **Test**: 핵심 비즈니스 로직(수율 계산, 상태 전이, 재고 판정)에 대한 단위 테스트 작성.
- **문서화**: `CLAUDE.md`(과제 명세 요약), `PRD.md`(본 문서) 최신 상태 유지.
- **Commit 이력**: 기능 단위로 의미 있는 커밋 분리.

## 8. 핵심 비즈니스 규칙 요약

| 규칙 | 계산식/조건 |
|---|---|
| 수율 | 정상 시료 수 / 총 생산 시료 수 |
| 재고 충분 판정 | 시료 재고 수량 ≥ 주문 수량 |
| 부족분 | 주문 수량 − 시료 재고 수량 (재고 부족 시) |
| 실 생산량 | `ceil(부족분 / 수율)` |
| 총 생산 시간 | 평균 생산시간 × 실 생산량 |
| 생산 큐 스케줄링 | FIFO (선입선출) |
| 재고 상태 표기 | 여유 / 부족 / 고갈(수량 0) |

## 9. 성공 기준 (Acceptance Criteria)

- [ ] 시료 등록 후 목록/검색에서 즉시 조회 가능
- [ ] 재고가 충분한 주문 승인 시 `CONFIRMED`로 전환되고 재고가 차감됨
- [ ] 재고가 부족한 주문 승인 시 `PRODUCING`으로 전환되고 생산 큐에 FIFO로 등록됨
- [ ] 생산 완료 시 실 생산량만큼 재고가 반영되고 상태가 `CONFIRMED`로 전환됨
- [ ] `CONFIRMED` 주문만 출고 처리 가능하며 처리 후 `RELEASE`로 전환됨
- [ ] 모니터링 메뉴에서 `REJECTED`를 제외한 상태별 주문 수와 재고 상태(여유/부족/고갈)가 정확히 표시됨
- [ ] 애플리케이션을 재시작해도 이전에 등록한 시료/주문 데이터가 유지됨
- [ ] 핵심 로직(수율 계산, 상태 전이, 재고 판정)에 대한 단위 테스트가 통과함

## 10. 가정 및 미결 사항

- 콘솔 화면 레이아웃은 자유롭게 구성 (PDF의 예시 UI는 참고용).
- 생산 라인은 단일 라인으로 가정하며, 생산 큐에 여러 주문이 대기 중이어도 한 번에 하나의 주문만 처리.
- 생산 시간의 실시간 경과 시뮬레이션 방식(실제 대기 시간 vs. 즉시 처리 후 진행률 표시)은 구현 단계에서 결정.
- 사용자 인증/권한 분리는 본 과제 범위에 포함하지 않음.
